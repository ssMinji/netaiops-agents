## [0:20] Metric Charts

These charts are all pulled from CloudWatch automatically — no manual dashboard setup. You open the page and the network state is right there. Just for reference, the traffic you see is generated by demo scripts running across the environment to keep the charts realistic.

## [0:45] Anomaly Report

Before we go into each agent, take a look at this. The Anomaly Detection Agent kicks off automatically as soon as you open the dashboard — no user query needed. It scans CloudWatch metric anomalies, VPC Flow Logs, and ELB status changes all at once.

This is the key point — 'Something is wrong, but we don't know what, where, or why.' This agent finds the what, where, and why automatically.

## [1:30] Anomaly Agent — Flow Log Deep Dive

**[Click "Discuss in Chat" → Flow Log Analysis]**

From the report, you can dive deeper by chatting with the agent. Flow Log analysis is one of the most time-consuming tasks in network operations — even in our 3-VPC environment, there are multiple log groups to correlate. The agent handles all of them in a single query.

Right now it's calling the actual CloudWatch Logs Insights API. This isn't a hardcoded dashboard — it dynamically analyzes based on whatever you ask.

## [3:00] Network Diagnostics Agent

**[Click "Network Diagnostics Agent"]**

Now the Network Diagnostics Agent. The Anomaly Agent analyzes traffic patterns — what's happening right now. The Network Agent is different — it inspects the infrastructure itself. VPC configurations, routing tables, security group rules. The Anomaly Agent tells you 'something abnormal is happening,' the Network Agent tells you 'here's what's misconfigured.'

It's now analyzing VPCs, subnets, routing tables, and security group rules — finding overly permissive security groups, unused ENIs, and configuration gaps.

## [3:45] Incident Agent

**[Click "Incident Analysis Agent"]**

While that runs — the Incident Agent. Designed for the customer running EKS at scale . It integrates Container Insights, OpenSearch logs, and GitHub. In production this would typically connect to Jira — we used GitHub Issues here to keep things simple.

## [4:00] Trigger + Agent Workflow

**[Click "CPU Spike Analysis"]**

Notice that the agent created a GitHub issue with severity labels, now it's pulling pod-level CPU metrics from Container Insights and cross-referencing with OpenSearch logs. Everything correlated automatically.

## [5:30] Incident Wrap + Closing

From trigger to root cause to a documented ticket — all automated. In this demo we triggered manually, but in production a CloudWatch alarm kicks this off automatically. Extend it with Jira and you get auto-assignment to the right on-call engineer, with root cause analysis pushed directly as a notification. SREs start from a hypothesis, not from zero.

We also have K8s and Istio agents — EKS cluster diagnostics, service mesh mTLS audits, canary deployments, latency hotspot detection. Same pattern, different layers.

Each agent is modular — deploy only what you need. Because everything runs on Bedrock AgentCore, there's no complex setup. Need new capabilities? Attach a Lambda function or API endpoint — the agent uses it immediately. Same platform, available to every AWS customer.

## [6:15] What We Learned

**[Switch to "What We Learned" slide]**

Three takeaways. First — invisibility is the real problem, not complexity. Both customers had complex systems, but nobody could see inside them. These agents make the invisible visible.

Second — AIOps starts in the mess. You don't need a clean environment first. Opaque traffic, scattered metrics — that's exactly what AIOps tackles first. The cleaner it gets, the faster the agents become.

Third — start when the customer is already asking 'where do we begin?' That's the signal.
